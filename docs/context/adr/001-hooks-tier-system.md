# ADR-001: Hooks 实践价值四层分级体系

- **状态**: 已采纳
- **日期**: 2026-04-30
- **背景**: Hooks 功能多样，需要系统性分级以便理解其价值和优先级

---

## 背景与问题

Qoder 支持多种 Hook 事件，每个 Hook 脚本的价值差异很大。需要一个分级框架来：
1. 判断哪些 Hook 优先落地
2. 评估新增 Hook 的必要性
3. 向其他人解释各 Hook 的作用层次

## 决策

采用四层分级体系（Tier 1-4），按价值密度排序：

| Tier | 名称 | 脚本 | 核心价值 |
|------|------|------|----------|
| T1 | 安全防护 | `security-gate.sh`、`prompt-guard.sh` | 防止不可逆损害，最高优先级 |
| T2 | 质量保障 | `auto-lint.sh`、`log-failure.sh` | 自动执行规范，减少人工审查 |
| T3 | 体验提升 | `notify-done.sh` | 改善工作流感受，不影响功能 |
| T4 | 知识积累 | `knowledge-trigger.sh` | 长期价值，短期可选 |

## 替代方案

- **不分级，按事件类型组织**：不直观，无法体现优先级
- **只分"必须/可选"两级**：过于粗糙，无法指导新 Hook 的位置

## 影响

- 新增 Hook 时先确定其 Tier，再决定是否落地
- T1 Hook 必须在任何项目中优先配置
- T4 Hook 的 CLI 限制不影响其价值定性
